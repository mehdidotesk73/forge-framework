# Forge Module System — Suite Developer Guide

This document is for **framework and Suite developers** working on
`packages/forge-py/`, `packages/forge-suite/`, or the Forge webapp. It covers
the architecture of the module system, what happens at each step of module
absorption, and what to consider when extending the feature.

---

## 1. Architecture Overview

### Integration point: `load_config`

The module system has exactly one integration point into the rest of the
framework: `load_config` in `packages/forge-py/forge/config.py`. When
`forge.toml` contains a `[[forge_modules]]` block, `load_config` imports
the module's `MODULE_CONFIG` object and merges its model/endpoint/pipeline
entries into `ProjectConfig` before returning. From that point on, every
other framework layer — `forge model build`, `forge endpoint build`,
`forge dev serve` — is completely unaware that these entries came from a
module rather than the project itself.

No other framework file knows about modules. This is deliberate.

### Key files

| File                      | Role                                                                                                                                          |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `forge/modules.py`        | Dataclass types: `ModuleConfig`, `ModelEntry`, `EndpointRepoEntry`, `PipelineEntry`. Imported by every `module.py` artifact.                  |
| `forge/config.py`         | `ForgeModuleConfig` (Pydantic model), field on `ProjectConfig`, merge loop in `load_config`, round-trip preservation in `save_config`.        |
| `forge/server/app.py`     | `load_endpoint_modules` — importlib fallback for pip-installed endpoint repos. `bootstrap_module_datasets` — idempotent dataset provisioning. |
| `forge/cli/dev_cmd.py`    | Calls `bootstrap_module_datasets(config, engine)` after `load_endpoint_modules`, before `create_app`.                                         |
| `forge/cli/module_cmd.py` | Click group: `add`, `remove`, `list`, `build`, `adopt`.                                                                                       |
| `forge/cli/main.py`       | Registers `module_group` as `forge module`.                                                                                                   |

### Module package layout

```
<module-dir>/
  forge.toml                ← standalone Forge project (runnable with forge dev serve)
  pyproject.toml            ← publishes forge-modules-<name> to PyPI
  pyrightconfig.json
  forge_modules/
    <snake_name>/           ← PEP 420 implicit namespace (NO forge_modules/__init__.py)
      __init__.py
      module.py             ← GENERATED by forge module build; committed; never hand-edited
  models/
  pipelines/
  endpoint_repos/
  apps/
  skills/
  .forge/artifacts/         ← pre-built schemas (committed)
  .forge/generated/         ← pre-built SDKs (committed)
```

**Namespace package**: there is intentionally no `forge_modules/__init__.py`.
Python's import system merges all installed `forge_modules.*` subpackages into
one virtual namespace at runtime (PEP 420). Multiple modules installed in the
same venv coexist without conflict.

### The `module.py` artifact

`module.py` is the published contract between a module package and any host
project. It declares exactly:

- Which model classes to import, and from which dotted path
- Which endpoint repo modules to load
- Which pipelines to register (stable UUIDs + display names)
- What dataset UUIDs the module owns (permanent — never change post-publish)

`forge module build` generates it by importing the model modules (to trigger
`@forge_model` decorators) and reading `forge.toml` for pipelines and endpoint
repos. It warns on UUID drift vs the previously committed `module.py`.

## 2. What Happens During Module Absorption (`forge module adopt`)

"Absorption" means taking a plain standalone Forge project and adding the
packaging layer that makes it a publishable pip module. Here is the exact
sequence:

### Step 1 — Copy (unless `--in-place`)

`shutil.copytree` from the source project into
`packages/forge-modules/<name>/`, skipping:

- `.venv/`, `node_modules/`, `dist/`, `.forge-suite/`
- The project-level `data/` directory
- `*.parquet`, `*.duckdb`, `*.duckdb.wal` files

The result is a clean copy of just the source code and config.

### Step 2 — Generate `pyproject.toml`

Creates a `[tool.setuptools.packages.find]` config that captures
`forge_modules*` (the namespace dir). Scans `endpoint_repos/` for
`anthropic`, `openai`, `boto3` imports and adds them as optional
dependencies if found.

### Step 3 — Create `forge_modules/<snake_name>/` namespace dir

Creates `forge_modules/<snake_name>/__init__.py`. Does **not** create
`forge_modules/__init__.py` — that would break the implicit namespace.

### Step 4 — Run `forge module build`

Invoked as a subprocess in the target directory. This:

1. Reads `forge.toml` for pipeline and endpoint repo entries
2. Adds the module dir to `sys.path`
3. Imports each model module — `@forge_model` decorators populate
   `_MODEL_REGISTRY` and `_CLASS_REGISTRY`
4. For each registered class, records `class_name`, `module`, `mode`,
   and `backing_dataset_id` from the decorator
5. Prefixes all `module` paths with `forge_modules.<snake_name>.` so they
   resolve as pip-installed paths (not project-local ones)
6. Writes `forge_modules/<snake_name>/module.py` from a template

**Critical**: The module paths written into `module.py` use the
`forge_modules.*` namespace. When the package is later pip-installed, these
are the importable paths. The original project-relative paths (`models.*`,
`pipelines.*`) only work while the project is run standalone.

## 3. What Happens When a Host Project Adds a Module (`forge module add`)

1. `pip install forge-modules-<name>` into the active venv
2. Import `forge_modules.<snake_name>.module` and verify `MODULE_CONFIG`
   is accessible
3. Append a `[[forge_modules]]` block to `forge.toml`:
   ```toml
   [[forge_modules]]
   name       = "<name>"
   package    = "forge-modules-<name>"
   config_var = "forge_modules.<snake_name>.module:MODULE_CONFIG"
   ```
4. Reload `ProjectConfig` — the merge loop in `load_config` runs and
   appends the module's models/endpoint_repos/pipelines to the in-memory
   config
5. Call `bootstrap_module_datasets(config, engine)` — for each UUID in
   `MODULE_CONFIG.dataset_ids`, if the dataset doesn't exist in
   `.forge/data/`, write an empty Parquet file and register it in DuckDB

After this, `forge model build`, `forge endpoint build`, and `forge dev serve`
all work as if the module's entries were hand-authored in the project.

## 4. Runtime Path Resolution: `load_endpoint_modules`

Project-local endpoint repos are at `<root>/endpoint_repos/<name>/`.
Pip-installed module endpoint repos are in site-packages as
`forge_modules.<name>.endpoint_repos.*`.

`load_endpoint_modules` handles both via a two-step check:

```
local_path = root / repo_cfg.module.replace(".", "/")
if local_path.exists():
    repo_path = local_path
    base_path = root          ← module_name derived from project root
else:
    spec = importlib.util.find_spec(repo_cfg.module)
    repo_path = Path(spec.submodule_search_locations[0])
    base_path = repo_path.parent  ← module_name derived from package root
```

`base_path` ensures the `module_name` string passed to
`importlib.import_module` is correct in both cases. If neither path
resolves, a warning is logged and the repo is skipped (never crashes).

## 5. Config Round-Tripping (`save_config`)

Any CLI command that rewrites `forge.toml` goes through `save_config`.
Without explicit handling, `tomli_w.dump` would silently drop
`[[forge_modules]]` blocks because they aren't in the standard
`ProjectConfig` field set.

`save_config` preserves them by appending:

```python
if config.forge_modules:
    data["forge_modules"] = [m.model_dump() for m in config.forge_modules]
```

**If you add a new top-level TOML section** to `ProjectConfig` in the
future, apply this same pattern — add a corresponding block in
`save_config`.

## 6. The `load_config` Merge Loop — Invariants

The merge loop runs **after** the `ProjectConfig` is constructed from the
project's own `forge.toml`. It appends — it does not deduplicate or
override. This means:

- A project can have a model with the same class name as a module model.
  Both will be present in `config.models`. The last one to register in
  `_CLASS_REGISTRY` wins at runtime.
- The loop uses `warnings.warn` on import failure, never `raise`. A missing
  module renders the associated entries absent but doesn't break unrelated
  commands.
- The loop imports at config-load time. This means any CLI command that
  calls `load_config` will attempt module imports. Keep `module.py`
  artifacts lightweight — they should contain no business logic.

The following are planned but not yet built. When implementing:

- Add `ForgeModule` snapshot model to `forge-webapp/models/models.py`
- Add `MODULE_DATASET_ID` placeholder + provision in `setup.sh`
- Add `ForgeModule` entry to `forge-webapp/forge.toml`
- Create `endpoint_repos/module_endpoints/` with `list_modules`,
  `sync_modules`, `add_module`, `remove_module`
- Create `ModulesPage.tsx` — two-panel: available modules / active modules
- Add "Modules" nav item to `Sidebar.tsx` (between Endpoints and Apps)

The `add_module` and `remove_module` endpoints should **delegate to the CLI
subprocess** (`forge module add/remove`) rather than calling operations
directly. This ensures pip install/uninstall runs in the correct project
venv, not the Suite venv.

See `.thoughts/forge-module-system-design.md` §4 for full spec.

## 8. UUID Stability Contract

Dataset UUIDs in `MODULE_CONFIG.dataset_ids` are **permanent** once a
version of the module is published. `forge module build` warns if any UUID
changed vs the existing `module.py`. A future hardening step should make
this an error.

The reason: if a published module changes a UUID, any host project that
has already bootstrapped the old UUID ends up with orphaned Parquet files
and a broken model reference. There is no migration path available to the
host project automatically.

## 9. What to Check When Extending the Module System

| Change                                          | What to verify                                                                            |
| ----------------------------------------------- | ----------------------------------------------------------------------------------------- |
| New field on `ProjectConfig`                    | Add to `save_config` or it will be silently dropped on rewrite                            |
| New entry type (e.g. `AppEntry`)                | Add to `ModuleConfig` dataclass, `load_config` merge loop, `module_cmd.py` build template |
| New CLI command                                 | Import in `module_cmd.py`, register under `module_group`                                  |
| Renaming `MODULE_CONFIG` convention             | Update `_default_config_var()` in `module_cmd.py` and the design doc                      |
| Changing `forge_modules.*` namespace convention | Update `module_build` path-prefixing logic and `_copytree_filtered`                       |
